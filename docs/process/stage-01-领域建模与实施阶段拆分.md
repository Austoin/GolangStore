# 阶段 01：领域建模与实施阶段拆分

## 时间
- 2026-03-24

## 新增上下文
- GitHub 仓库归属：`github.com/austoin`
- GitHub 仓库名：`github.com/austoin/GolangStore`

## 数据模型决策
- MySQL 核心表保持最小集合：
  - `users`
  - `products`
  - `product_stocks`
  - `cart_items`
  - `orders`
  - `order_items`
- 秒杀场景可按实现复杂度决定是否单独增加 `seckill_products`，当前先作为可选项预留，不强制进入一期第一批建表。

## 缓存模型决策
- Redis 只承担三类职责：
  - 热点商品缓存
  - 秒杀库存原子扣减
  - 短期订单状态缓存
- MySQL 仍为最终持久化真相源。
- Redis 不承担永久业务真相，不做业务数据长期双写主存储。

## 秒杀链路决策
- 入口链路：`gateway -> order-service -> Redis Lua -> MySQL`
- 执行顺序：
  1. 接收秒杀下单请求。
  2. 调用 Redis Lua 脚本原子判断并扣减库存。
  3. 扣减成功后写入订单与订单项。
  4. 若 MySQL 落库失败，则执行库存回补。
- 当前不引入独立 MQ，先保证原子扣减、防超卖、落库回补三个核心能力成立。

## 实施阶段拆分

### 阶段 1：仓库骨架与公共基础设施
- 目标：建立 monorepo 基础目录、公共配置、日志、MySQL/Redis 初始化能力。
- 主要产物：
  - `cmd/`
  - `internal/`
  - `pkg/`
  - `configs/`
  - `docker-compose.yml`
  - 基础 `Makefile`
- 验收标准：
  - 项目目录结构稳定。
  - 所有基础配置可本地加载。
  - MySQL/Redis 连接初始化具备单元测试或最小可验证入口。

### 阶段 2：商品、购物车、订单主链路
- 目标：先打通普通交易闭环。
- 主要产物：
  - 商品列表/详情/库存查询接口
  - 购物车增删改查接口
  - 创建订单、订单详情、订单状态查询接口
- 验收标准：
  - 可完成“查商品 -> 加购物车 -> 下单 -> 查订单”的完整链路。
  - MySQL 表结构与最小种子数据可用。

### 阶段 3：Redis 库存与秒杀链路
- 目标：建立高并发库存保护层。
- 主要产物：
  - Redis 热点库存缓存
  - Lua 扣减脚本
  - 秒杀下单路径
  - 落库失败后的库存回补逻辑
- 验收标准：
  - 并发测试下无超卖。
  - Redis 与 MySQL 的职责边界明确。

### 阶段 4：MCP 网关与 LLM Skills
- 目标：将商城核心能力暴露为 MCP 可调用工具。
- 主要产物：
  - `mcp-gateway`
  - `库存查询` skill
  - `订单状态查询` skill
- 验收标准：
  - MCP client 能发现工具。
  - MCP client 能成功调用库存查询与订单状态查询。

### 阶段 5：Docker Compose 联调与交付整理
- 目标：提供本地一键联调环境与交付说明。
- 主要产物：
  - 服务 Dockerfile
  - Compose 编排
  - 初始化 SQL
  - 运行说明文档
- 验收标准：
  - 本地可一键启动 MySQL、Redis 与全部服务。
  - 核心接口与 MCP 链路可 smoke test。

## 执行顺序约束
- 必须先完成阶段 1，再进入业务实现。
- 阶段 2 未闭环前，不进入阶段 4。
- 阶段 3 必须在普通订单链路稳定后再接入。
- 阶段 5 只在前 4 个阶段最小闭环成立后统一整理。

## 风险
- 如果在阶段 2 前提前引入 MCP、K8s、后台权限，会打乱主链路建设顺序。
- 如果在阶段 3 前没有先稳定普通下单模型，秒杀链路会被迫承载过多基础职责。

## 阶段结论
- 已完成领域模型与高层阶段拆分。
- 已明确数据边界、缓存边界和实施优先级。
- 可以进入项目骨架创建阶段。
